## üß± BUILDER INTELLIGENCE

**Methodology beat:** üî•

Two research findings published today crystallize something important for your `cockpit-core.mdc` work:

LLM-generated context files (from `/init`) reduced task success by 2‚Äì3% while increasing cost by over 20%. Developer-written files improved success by about 4% ‚Äî but also increased cost by up to 19%. The delta is *what's in the file*, not whether it exists. The practical filter: can the agent discover this on its own by reading your code? If yes, delete it.

The anchoring problem is real: if your AGENTS.md mentions something ‚Äî even in passing ‚Äî the model has it in context for every prompt. LLMs don't distinguish between "this is what we used to do" and "this is what you should do." Every stale line in your `.mdc` is actively misleading the agent, not just wasting tokens.

On the chain enforcement problem you're actively fighting: a single monolithic file gets loaded the same way every time ‚Äî a flat instruction set can't condition on what kind of task is being run. This explains *precisely* why `.mdc` rules alone don't enforce your Superpowers chain. The file doesn't know if you're in planning mode or execution mode. Your `cockpit-repass` challenge gates are the right architectural response to this.

Simon Willison's new Agentic Engineering Patterns guide also surfaces the TDD angle: "Red/green TDD describes how test-first development helps agents write more succinct and reliable code with minimal extra prompting." Your Design Challenge gate is essentially a pre-flight version of this.

**Superpowers v4.3.0 note:** A critical bug was fixed on Feb 12 ‚Äî the `SessionStart` hook was running asynchronously, which silently dropped the `using-superpowers` bootstrap context before the agent's first turn.
