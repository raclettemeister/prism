## 🧱 BUILDER INTELLIGENCE

**Methodology beat:**

The emerging pattern is **spec-driven development as the primary AI workflow.** The argument: specs become the source of truth, code becomes an artifact of the spec. You review the steps and acceptance criteria — not 500-line diffs. Human-in-the-loop approval moves from "Did you write this correctly?" to "Are we solving the right problem with the right constraints?"

The practical architecture proposed has four layers: (1) run multiple agents and pick the best output, (2) deterministic guardrails (tests, type checks — things that don't have opinions, just facts), (3) humans define acceptance criteria in natural language before the first line of code is generated, and (4) narrow permission scoping per agent task.

**This is directly relevant to how you use Cursor.** Your existing skill system (blog pipeline, business status updater, etc.) is already spec-adjacent — but are you writing acceptance criteria *before* you prompt, or reviewing output after? That delta matters now.

Simon Willison also published a concrete example yesterday: he built a GIF optimization tool using WebAssembly and Gifsicle by prompting Claude Code from his iPhone, letting it handle the entire compilation, UI, and testing loop autonomously — including fixing bugs it found itself via browser automation with Rodney. The signal: the prompt-from-phone, agent-handles-everything pattern is production-ready today.
